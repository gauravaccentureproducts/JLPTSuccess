You are operating in implementation mode on the JLPT N5 sub-app of
JLPTSuccess. Your single job in this run is to transition the app
from a 5-locale shell (English / Vietnamese / Indonesian / Nepali /
Chinese) to a 2-locale shell (English / Hindi) — completely,

You WILL implement this end-to-end: code, content data, docs,

## STRATEGIC CONTEXT (read this first)

The 5-locale shell (en/vi/id/ne/zh) was originally shipped to
support a "multilingual non-English-native learner" niche claim
(niche-N1 in the audit prompt N5Improvement.txt). The 2026-05-06
deep market research (see commit history + xlsx audit registry +
the conversation that produced this prompt) found that:

- Vietnamese, Indonesian, Burmese, Bengali, Sinhala, Brazilian
  Portuguese, Filipino/Tagalog, Thai, and Chinese all have at
  least one — usually several — established competing JLPT prep
  apps in their native language. The "no competitor" framing for
  niche-N1 was wrong for those locales.
- Nepali is genuinely uncovered, but the absolute market is too
  small (~2-5K JLPT/yr) to be the strategic primary.
- **Hindi is the unique high-demand-low-competition gap.** India
  is the 5th-largest JLPT country worldwide; ~50K applicants/year;
  73% are at N5 or N4 level (perfect product-market fit). NO
  dedicated Hindi-native JLPT prep app was found in app-store or
  curated-list searches. Yoisho Academy delivers in English with
  optional Hindi tutoring — closest competitor, but not an app
  and not Hindi-medium pedagogy.

The strategic decision is: **stop spreading thin across 5 mediocre
locale shells; commit to 2 locales (English + Hindi) and ship
those at native-quality depth.** Niche-N1 is reframed from
"multilingual non-English-native" to "the only privacy-first
no-account offline JLPT app with English + native Hindi pedagogy."

This is a strategic narrowing, not a feature regression. Existing
users with persisted vi/id/ne/zh locale settings will fall back
gracefully to English. No user loses access to any content.

## GOAL — END STATE

1. `N5/locales/` contains exactly two files: `en.json` and `hi.json`.

   `{en, hi}`. Any other persisted or requested locale falls back

### Phase 2: Add migration code for existing users

Goal: when a user with `localStorage.locale ∈ {vi, id, ne, zh}`

migration code goes in BEFORE the removal so it ships ahead of

Tasks:
- Identify the exact storage key and module that persists locale.
  Likely candidates: `js/storage.js` (with helper
  `getSettings()` / `setSettings()`), `js/i18n.js` (with bootstrap
  code), or `js/app.js` (with a one-time hydrate step).
- Add a migration function `migrateLocaleSetting()` that:
  1. Reads the persisted locale from localStorage.
  2. If the value is in `{vi, id, ne, zh}`, sets it to `'en'`
     and persists immediately.
  3. If the value is anything not in `{en, hi}` (including null,
     undefined, junk), defaults to `'en'`.
  4. If the value is already in `{en, hi}`, no-op.
  5. Logs a console.info exactly once per session for telemetry-
     free observability: `"locale migrated: vi → en"` (or
     equivalent). This is dev-only signal; no PII, no network.
- Call `migrateLocaleSetting()` from app bootstrap, BEFORE any
  i18n-dependent rendering code runs.
- Add a unit test (or extend the existing tests.html harness)
  that simulates each of the 4 deprecated locales in localStorage
  and asserts the migration result is `'en'`.
- Smoke test: in browser DevTools, set
  `localStorage.setItem('jlpt-n5-tutor:settings', JSON.stringify({locale: 'vi', ...}))`,

  the console.
- Commit message: `feat(i18n): migrate deprecated locales (vi/id/ne/zh) to en`.

### Grammar L1-interference notes (Hindi-specific)

Replace the existing
Vietnamese / Indonesian / Nepali / Mandarin L1 notes (from
N5Improvement.txt's grammar pedagogical-depth section) with
Hindi-specific notes. Mandatory coverage:

- **SOV word order shared advantage.** Hindi and Japanese are
  both subject-object-verb. Hindi speakers transfer this directly
  and have a structural advantage over English-L1 learners.
  Note this as a confidence-builder, not a pitfall.

- **Postposition vs particle mapping (the most important Hindi
  L1 note).** Hindi has postpositions (ने/को/से/में/पर/के लिए);
  Japanese has particles (が/を/から/に/で/へ/と/も). Mapping is
  not 1:1 — common confusion:
  - Hindi से ("from" / "by means of") splits into Japanese から
    (source) AND で (instrument). Hindi speakers often pick the
    wrong one.
  - Hindi को (direct object marker + indirect object dative)
    splits into Japanese を (direct object) AND に (indirect
    object / direction). Same pitfall.
  - Hindi में ("in") usually maps to Japanese に (existence
    location) but can map to で (action location) — context-
    dependent. Hindi speakers benefit from explicit に vs で
    teaching with the existence-vs-action distinction.

- **Verb agreement (gender + number).** Hindi conjugates verbs
  for person, number, AND gender. Japanese does not. Hindi
  speakers may unconsciously look for an agreement marker that
  doesn't exist; this is harmless but a teaching opportunity to
  point out "Japanese verbs don't change for who".

- **Tense over-marking.** Hindi has more tense forms (perfect,
  habitual, progressive, past habitual) than Japanese, which
  uses just present/non-past + past + a few aspect markers
  (〜ている / 〜たことがある). Hindi speakers may try to
  over-mark tense by stacking aspect markers. Note specifically
  that 〜ている covers BOTH "is doing" and "has been doing"
  contexts.

- **Politeness mismatch (3-tier Hindi vs morphological Japanese
  honorifics).** Hindi has a 3-tier pronoun politeness (तू / तुम
  / आप) but minimal verb morphology for politeness — politeness
  is mostly expressed lexically. Japanese has 2-tier basic
  politeness (casual / です・ます) PLUS a separate full
  morphological honorific system (謙譲語 / 尊敬語). Hindi
  speakers benefit from explicit teaching that Japanese keigo
  is a verb-morphology system, not a vocabulary swap.

- **Negative formation placement.** Hindi नहीं sits before the
  verb or main predicate; Japanese ない attaches to the verb
  stem. This is a placement difference, not a meaning difference,
  but Hindi speakers may produce "verb + nahi" calques in
  Japanese.

- **Question particle placement.** Hindi क्या at sentence start
  vs Japanese か at sentence end. Hindi speakers often initially
  omit か because the rising intonation alone (which works in
  Hindi for some questions) feels sufficient.

- **Plural marking.** Hindi marks plurals (लोग, बच्चे, etc.).
  Japanese mostly doesn't (人 = "person" or "people" depending
  on context). Hindi speakers initially over-pluralize using
  〜たち / 〜ら beyond the (limited) cases where it's natural.

- **Counter-word system overlap with mismatch.** Both Hindi and
  Japanese have classifier/counter words but the systems differ
  significantly. Hindi has fewer counters and uses them less
  obligatorily than Japanese. Hindi speakers benefit from
  counter-word drilling specifically because the concept is
  familiar but the inventory is unfamiliar.

These notes go into `N5/data/grammar.json` per-pattern
`notes.l1_hindi` (or whatever schema field the data model uses;
preserve the existing schema conventions). Each pattern that
involves any of the above contrasts should carry the relevant
note.
