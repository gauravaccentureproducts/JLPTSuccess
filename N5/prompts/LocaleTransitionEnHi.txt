## THE NATIVE-HINDI-SCHOLAR QUALITY RUBRIC

A native Hindi scholar reviewing a JLPT-pedagogical Hindi gloss
applies these tests in order. Any failed test = flag for rewrite.

The rubric was validated through cycle 1; cycle 2 should expect
to see most R-1, R-3, R-7 issues already resolved and focus on
R-2, R-4, R-5, R-6.

### R-1. Script integrity (mechanical, mostly resolved in cycle 1)

- The string contains at least one Devanagari character
  (Unicode U+0900–U+097F) UNLESS the slot is documented allowlist.
- No mojibake, no replacement characters (U+FFFD).
- Half-form / half-letter consonants render correctly.
- Nukta usage is correct (क़/ख़/ग़/ज़/फ़/ड़/ढ़ only when
  source phoneme demands).

**R-1.1 (NEW from cycle 1 learnings) — Kana-prefix convention.**
Japanese grammatical particles attached to Hindi terms MUST be
in Japanese script (hiragana), not Latin romaji or Devanagari
transliteration. The convention applies symmetrically across
EN and HI prose:
  CORRECT:   `な-विशेषण`, `い-विशेषण`, `て-रूप`, `た-रूप`,
             `ない-form`, `ます-form`, `たい-form`
  WRONG:     `na-विशेषण` (Latin), `ना-विशेषण` (Devanagari xliteration)
  WRONG:     `na-adjective`, `te-form` in English prose
  RIGHT:     `な-adjective`, `て-form` (kana + English word)

This rule was missing from the original cycle-1 prompt and only
surfaced via reviewer follow-up. It now has a dedicated CI
invariant (target: JA-NN, cycle 2).

**R-1.2 (NEW from cycle 1) — Hindi tokens in Hindi prose stay
Devanagari.** Hindi-language references inside Hindi explanations
must NOT be Latin-romanized. Example caught in cycle 1:
  WRONG:     `te-form की tha tha का त-रूप`
             ("tha tha" = था-था, the Hindi past-tense marker)
  RIGHT:     `て-form की था-था का त-रूप`

### R-2. Naturalness (most important; CYCLE-2 PRIMARY FOCUS)

- Read the Hindi sentence aloud. Would a native-Hindi-speaking
  Indian Hindi-medium learner produce this sentence
  spontaneously, or does it sound like a Google Translate output
  / a calque from English?
- Word order should be natural Hindi (SOV, postpositional). A
  literal English-to-Hindi rendering with English word order
  (SVO) is a fail.
- Cycle-1 mechanical translation produced many sentences with
  English-style word order (e.g., "ます विनम्र अभूत-सकारात्मक रूप है"
  follows EN order; native Hindi might phrase it as
  "विनम्र अभूत-सकारात्मक रूप ます है" or restructure entirely).
- Idiomatic verb phrases > literal verb phrases.

### R-3. Register match

- Vocab gloss is dictionary-short (1-3 words); Hindi gloss
  matches.
- Grammar explanation_hi is teaching prose: respectful तुम/आप-
  level. NOT casual तू/तेरा.
- L1 notes: encouraging + concrete. Avoid linguist-grade jargon.

### R-4. Vocabulary appropriateness for मानक हिंदी

- Prefer मानक हिंदी default. Persian/Arabic OK when colloquial.
  Avoid heavy Urdu register.
- Sanskrit-origin OK for academic concepts; avoid Sanskritized
  over-formal vocabulary for everyday concepts.
- For tech loanwords with no settled Hindi: Devanagari
  transliteration of the English (कंप्यूटर, टेलीफ़ोन, काउंटर).

### R-5. Grammatical accuracy

- Gender agreement on every adjective + verb in habitual/past.
- Postposition placement grammatical (ne/ko/se/me/par/ke-liye).
- Compound verbs (light verbs) usage natural.
- Number agreement plural-clean.

### R-6. JLPT-pedagogical fidelity

The Hindi explanation must be FAITHFUL to the Japanese it explains:
- A Japanese verb conjugation explanation in Hindi must not
  introduce errors about the Japanese form.
- L1-interference notes must compare correct Hindi to correct
  Japanese on BOTH sides. Comparing wrong Hindi to right
  Japanese is worse than English-only would have been.
- Cycle 1 caught 1 instance: n5-029 used "सोचना का" (broken
  Hindi grammar) to illustrate Japanese の. Cycle 2 should
  spot-check L1 notes specifically for similar issues.

### R-7. Consistency across the corpus

- Same Japanese vocab → same Hindi gloss across vocab/kanji/
  questions/reading.
- Same grammar pattern → same Hindi pattern name across surfaces.
- Pronoun choice (तुम vs आप) consistent.

## COMMON GOTCHAS (learned the hard way in cycle 1)

These tripped me up; document them so cycle 2 doesn't repeat:

### Field naming inconsistency
- Questions use `explanation_en` (NOT `explanation`).
- Some entries use `explanation`, some use `explanation_en`,
  some have BOTH. Cycle-1 sample script missed half the
  placeholders by checking only one field name.
- ALWAYS try both:
  `q.get('explanation_en') or q.get('explanation') or ''`

### JSON-escaped quotes break Edit
- The Edit tool's `old_string` matches the raw bytes in the
  file, NOT the visually-rendered text.
- A JSON file storing `"text with \"quoted\" stuff"` requires
  Edit's `old_string` to include `\"` literally.
- If Edit fails with "String to replace not found" but you
  can see the string with Grep, check for `\"` in the file
  vs unescaped `"` in your old_string.

### sys.stdout side-effects break imports
- Many cycle-1 scripts have
  `sys.stdout = io.TextIOWrapper(sys.stdout.buffer, ...)` at
  module-level for cp932-codec safety.
- This breaks if you import the module from another script
  (the wrap closes the original stdout, then a 2nd import
  tries to wrap a closed file).
- Fix: use the cycle-1 pattern of exec'ing source via
  `compile() + exec()` after stripping the sys.stdout line:
    src = path.read_text(encoding='utf-8')
    src = re.sub(r'^sys\.stdout\s*=.*$', '', src, flags=re.MULTILINE)
    src = re.sub(r"if __name__ == '__main__':[\s\S]*$", '', src)
    src = re.sub(r"def main\(\):[\s\S]*?(?=\n\S|\Z)", '', src)
    ns = {'__file__': str(path), '__name__': '_loaded'}
    exec(compile(src, filename, 'exec'), ns)
  See fix_hi01_paper_rationale_hi_2026_05_07.py:load_rules.

### ROOT path resolution after archive moves
- Diagnostic scripts originally lived in `tools/` but were
  moved to `not-required/tools-archive/`. The original
  `parent.parent` ROOT calc broke after the move.
- Fix (cycle-1 pattern): walk up to find data/:
    ROOT = Path(__file__).resolve().parent
    while not (ROOT / 'data').exists() and ROOT != ROOT.parent:
        ROOT = ROOT.parent
- Apply this pattern to ALL new scripts; future archive
  moves won't break them.

### Phase ordering: re-translate THEN re-sweep
- Phases F+G translate placeholders/coverage gaps from English
  to Hindi via glossary.
- The translated output INTRODUCES new English residuals
  (words my glossary doesn't cover).
- Phase A romanization sweep should re-run AFTER F+G to catch
  those residuals.
- Cycle 1 missed this and had to spot-fix manually. Cycle 2:
  bake the re-sweep into Phases 1-2.

### Mechanical translation quality ceiling
- A 250-rule glossary gets ~70% Hindi-quality coverage
  mechanically.
- Single-word residuals (predicate, ongoing, request) are
  high-value to fix; sentence-level naturalness needs human.
- Provenance honesty is critical: mechanical output is
  llm_curated, NEVER native_reviewed.

### Linter race conditions during multi-edit batches
- Some IDE / formatter processes auto-modify files between
  Read and Edit calls.
- The Edit tool errors with "File has been modified since read".
- Fix: re-Read before re-Edit. The harness tracks file state.

### CI invariant count drift
- Cycle 1 started with 48 invariants; JA-40 was added
  externally during the session, ending at 49.
- Cycle 2: check the invariant count at start; the next free
  number is `total + 1`, not a hardcoded "JA-40".

### Auto-archive of new scripts
