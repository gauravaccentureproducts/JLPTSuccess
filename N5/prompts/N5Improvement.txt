**Grammar pedagogical depth (per-pattern audit)**

The benchmark for N5 grammar is the union of:
  - JLPT.jp official 旧出題基準 (1994/2002, ~80-90 patterns at "level 4")
  - Genki I (Lessons 1-12, ~120 patterns)
  - Minna no Nihongo I + II first half (~150 patterns)
  - JLPT Sensei N5 list (~135 patterns)
  - Bunpro N5 deck (currently the de-facto reference at ~140 patterns)

The app currently ships **178 patterns** (per `data/grammar.json`).
For each pattern, the audit must check:

- **Coverage and tagging.** Does the pattern appear in the JLPT.jp
  official scope, in Genki, in Minna, in Bunpro, in JLPT Sensei?
  Each pattern should carry a `sources` array listing which canonical
  references include it (to support content provenance and to flag
  patterns that are over-extended into N4+).
- **Tier classification.** `core_n5`, `late_n5`, `n4_borderline` —
  is the tier correct? A pattern incorrectly tagged `core_n5` when
  it's actually N4 misleads beginners.
- **Form rules — depth.** Does `form_rules.attaches_to` cover ALL
  legal attachment surfaces (verb-stem, verb-te, verb-nai, verb-ta,
  verb-plain, verb-volitional, na-adj root, na-adj root + な,
  i-adj root, i-adj-ku form, noun, noun + の, plain clause)?
  Does `form_rules.conjugations` include the affirmative + negative +
  past + past-negative paradigm where applicable?
- **Examples — count, breadth, and authenticity.** ≥3 example
  sentences per pattern is the floor; 5-7 is best-in-class (Bunpro
  ships 5-7 per pattern). Each example must:
  - Use only N5 vocab + kanji (verifiable per JA-13 / JA-1).
  - Demonstrate a different attachment surface or register from
    the others (i.e. not 5 paraphrases of the same sentence).
  - Include a `vocab_ids` array (JA-17) so renderers can deep-link
    to the per-word page.
  - Cover the most common contexts a learner will actually meet
    (school, home, food/restaurant, weather, transport, family).
- **Common mistakes — must be present, must be specific.** For each
  pattern, ≥1 `common_mistakes` entry showing a wrong sentence and
  the corrected form, with the *why*. Generic "pay attention to
  conjugation" is not acceptable. Specific "Beginners often write
  〜ては instead of 〜ても when the meaning is concessive ('even
  if')" is.
- **Contrasts — adjacent pattern disambiguation.** For confusable
  pairs, an explicit `contrasts.with_pattern_id` cross-link with a
  one-paragraph "use X when ... use Y when ...". The mandatory
  contrast set at N5:
  - は vs が (topic vs subject)
  - から vs ので (causal — register)
  - も vs と (additive vs comitative)
  - で vs に (location of action vs location of existence)
  - けど / けれど vs が (conjunctional contrast — register)
  - 〜たことがある vs 〜た (experience vs past — temporal-marker rule)
  - 〜ている (progressive vs resultative)
  - て-form vs て-form + います (chained vs continuous)
  - 〜たい vs 〜ほしい (first-person desire vs third-person)
  - 〜ましょう vs 〜ませんか (proposal vs invitation)
  - あげる/くれる/もらう trio (giving direction)
- **Register and politeness.** Each pattern must be tagged with the
  register it belongs to: `casual`, `polite (です・ます)`, `humble
  (謙譲)`, `respectful (尊敬)`. Mixing within one example without
  pedagogical reason is a finding.
- **Pitch accent on examples.** Pitch accent isn't a JLPT scoring
  criterion but is a strong learner trust signal. Items that include
  it (or expose it via a kanji-popover) signal seriousness.
- **L1-interference notes — Hindi (niche-N1 unlock).** For each
  pattern, flag mistakes specifically common to Hindi-L1 learners.
  The 9 mandatory Hindi-specific contrast areas an audit must look
  for evidence of:
  - **SOV word-order shared advantage.** Both Hindi and Japanese
    are subject-object-verb. Hindi speakers transfer this directly
    and have a structural advantage over English-L1 learners. Note
    this as a confidence-builder per pattern, not a pitfall.
  - **Postposition vs particle mapping.** The single most important
    Hindi L1 note across N5 grammar:
    - Hindi से ("from" / "by means of") splits into Japanese から
      (source) AND で (instrument). Hindi speakers often pick the
      wrong one.
    - Hindi को (direct + indirect object marker) splits into
      Japanese を (direct) AND に (indirect / direction).
    - Hindi में ("in") usually maps to Japanese に (existence
      location) but can map to で (action location) — context-
      dependent. Explicit に-vs-で teaching with the
      existence-vs-action distinction is required.
  - **Verb agreement (gender + number).** Hindi conjugates verbs
    for person, number, AND gender. Japanese does not. Hindi
    speakers may unconsciously look for an agreement marker that
    doesn't exist; teaching point: "Japanese verbs don't change
    for who."
  - **Tense over-marking.** Hindi has more tense forms (perfect /
    habitual / progressive / past habitual) than Japanese, which
    uses present/non-past + past + a few aspect markers. Hindi
    speakers may stack aspect markers; specifically 〜ている
    covers BOTH "is doing" and "has been doing" contexts.
  - **Politeness mismatch (3-tier Hindi vs morphological Japanese
    keigo).** Hindi has 3-tier pronoun politeness (तू / तुम / आप)
    but minimal verb morphology for politeness. Japanese has
    2-tier basic (casual / です・ます) PLUS a separate full
    morphological honorific system (謙譲語 / 尊敬語). Teaching
    point: Japanese keigo is verb-morphology, not vocabulary swap.
  - **Negative formation placement.** Hindi नहीं sits before the
    verb; Japanese ない attaches to the verb stem. Placement
    difference, not meaning — Hindi speakers may produce
    "verb + nahi" calques.
  - **Question particle placement.** Hindi क्या at sentence start
    vs Japanese か at sentence end. Hindi speakers often initially
    omit か because rising intonation alone (which works in some
    Hindi questions) feels sufficient.
  - **Plural marking.** Hindi marks plurals (लोग, बच्चे). Japanese
    mostly doesn't. Hindi speakers initially over-pluralize using
    〜たち / 〜ら beyond the limited cases where it's natural.
  - **Counter-word system overlap with mismatch.** Both Hindi and
    Japanese have classifier/counter words but the systems differ
    significantly — Hindi has fewer counters and uses them less
    obligatorily. Hindi speakers benefit from explicit counter-
    word drilling because the concept is familiar but the
    inventory is unfamiliar.
  These notes are the strongest single content lever for niche-N1
  market positioning under the Hindi-specific framing. Absent or
  generic L1 notes is a major finding.

**Vocabulary pedagogical depth (per-word audit)**

The benchmark for N5 vocabulary is ~700-900 entries depending on
source (JLPT.jp official scope ~700; Genki I ~900; Minna I+II partial
~1200; JLPT Sensei ~700). The app currently ships **1041 entries**.
For each entry, the audit must check:

- **Coverage and tagging.** Does the word appear in the JLPT.jp
  official list? In Genki? In Minna? In Tanos? Each entry should
  carry a `sources` array. Entries not in any canonical source are
  candidates for either `[Cul]` (cultural — useful but not exam) or
  `[Ext]` (extended — borderline N4) or removal.
- **Per-word fields — depth.** A serious N5 entry has:
  - `form` — the word as written (kanji + okurigana, or kana-only).
  - `reading` — kana reading (essential).
  - `pitch_accent` — Tokyo dialect pitch pattern (e.g. 「LH」 for
    flat-rising, with downstep position). Currently absent? That's
    a niche-N1 unlock for serious learners.
  - `pos` — part of speech, sub-classified:
    - Verbs: godan / ichidan / irregular (来る / する).
    - i-adj vs na-adj (a frequent learner confusion).
    - Transitivity: transitive (他動詞) / intransitive (自動詞).
    - Counter class for nouns that take counters (本 takes さつ,
      cars take 台, etc.).
  - `gloss` — English meaning(s) ranked by frequency.
  - `gloss_translations.hi` — Devanagari Hindi translation of the
    English gloss. Currently absent on most/all entries? That is
    THE primary niche-N1 gap and the largest single depth-deficit
    in the corpus (1041 entries × 1 missing locale = 1041 deficit
    rows). Expected coverage target: 100% en, ≥ 95% hi.
  - ≥2 example sentences per word, each using only N5 scope.
  - `collocations` — common companion words. E.g. for 雨 (rain):
    雨が降る、雨が止む、雨に濡れる、雨宿り.
  - `register` tag where applicable — humble (拝見), respectful
    (ご覧になる), formal (本日 vs 今日), casual (〜じゃん).
- **Transitivity pair adjacency.** Every transitive verb must be
  cross-linked to its intransitive partner where one exists. The
  audit must verify the canonical N5 pairs:
  - 開ける / 開く, 閉める / 閉まる, 入れる / 入る, 出す / 出る,
    始める / 始まる, 止める / 止まる, つける / つく, 消す / 消える,
    起こす / 起きる, 落とす / 落ちる, 直す / 直る, 切る / 切れる.
  Mirror-pair adjacency in the catalog and explicit "see also" links
  in the per-word page.
- **Verb-class flag exceptions.** The Group-1-exception verbs
  (X-6.6 invariant) — verbs that LOOK like Group-2 but conjugate as
  Group-1 — must each carry the explicit "Group 1 exception"
  annotation:
  - 入る (はいる), 帰る (かえる), 走る (はしる), 知る (しる),
    切る (きる), 要る (いる).
- **Counter-noun pairing.** Nouns that take a specific counter must
  list it. 本 → 冊, cars → 台, animals → 匹, flat objects → 枚, etc.
- **Honorific / humble register chains.** N5 has a small set:
  いる⇄いらっしゃる⇄おる, 食べる⇄召し上がる⇄いただく,
  見る⇄ご覧になる⇄拝見する, etc. These should be cross-linked
  in both directions.
- **False-friend flags (Hindi-specific framing).** Hindi speakers
  do NOT share kanji with Japanese, so the legacy
  shared-script false-friend trap (the way Mandarin learners
  conflate 勉強 with 勉强) is largely absent for the Hindi-L1
  audience. The dominant false-friend risk for Hindi speakers
  is structural / phonological, not lexical:
  - Phonological collisions (a small set): Japanese サ-syllabary
    words occasionally collide with Sanskrit-origin Hindi
    vocabulary in surface phonology but not meaning. Audit
    should flag any genuinely confusable pair if it emerges,
    not invent ones.
  - **Kanji-cognate notes for learners with prior Chinese
    exposure** — keep these as cross-cutting English-language
    pedagogical notes (helpful for Indian learners with prior
    business-Chinese or anime-Chinese exposure, and for English-
    L1 learners generally), NOT as Hindi-locale-specific
    `gloss_translations.hi` content:
    - 勉強 (study) ≠ Chinese 勉强 (reluctantly)
    - 大丈夫 (OK) — literal kanji-only reading suggests "big
      husband", which is wrong; the idiom means "OK / fine"
    - 手紙 (letter) ≠ Chinese 手纸 (toilet paper)
    - 娘 (daughter) ≠ Chinese 娘 (mother)
  These appear in English-language explanatory notes; they are
  NOT translated into Hindi as locale-specific content.

- **Mnemonic / radical decomposition.** WaniKani's defining
  advantage is its radical-based mnemonics ("the eye 目 sees the
  tree 木 — 相 'mutual'"). The app currently has none. This is the
  single largest UX gap vs WaniKani for the kanji surface.
  Even minimal radical decomposition (1-2 sentences per kanji) is
  better than nothing. Could be sourced from KanjiVG decomposition
  + LLM-curated, then native-reviewed.
- **Confusable kanji clusters.** The audit must verify that
  visually-confusable N5 kanji are explicitly linked to each other
  with a "compare" callout:
  - 大 / 犬 / 太 (the dot moves)
  - 木 / 本 / 末 / 未 (line position)
  - 人 / 入 / 八 (radical similarity)
  - 日 / 目 / 白 (rectangle variations)
  - 千 / 干 / 王 / 玉 (vertical bar)
  - 上 / 止 / 正 (cross-bar position)
  - 古 / 占 (stroke count)
  - 千 / 午 (top stroke direction)
  Without these explicit cross-links, learners conflate them.
- **Stroke-order common mistakes.** A small set of N5 kanji have
  textbook stroke-order traps (e.g. the order of horizontal vs
  vertical strokes in 田, the direction in 力, the order in 必).
  Surfacing these as warnings is a content-depth lever.
- **Search / filter — power-user features.** By: reading
  (kana-prefix and kana-substring), meaning (English-substring),
  stroke count (range), radical, frequency rank, recognition
  priority, presence in user's `knownKanji` set.
- **Audio coverage consistency.** Each example sentence's audio
  exists in `audio/...`. Each on/kun-yomi reading should also have
  audio (or be derivable from the example audio).

**Reading / Dokkai pedagogical depth (N5-specific)**

The official JLPT N5 dokkai paper is structured as:
- **Mondai 4 (内容理解 — 短文)**: 100-200-character passages,
  3 questions across 3 passages = 9 questions.
- **Mondai 5 (内容理解 — 中文)**: 250-300-character passages,
  2 questions across 1-2 passages = 4 questions.
- **Mondai 6 (情報検索)**: Information search — timetable, poster,
  notice — 2 questions.

The app currently ships **40 reading passages**. For each, audit:

- **Mondai-type tagging.** Each passage carries `mondai` ∈ {4, 5, 6}
  and `format_type` from a closed enum (per JA-29). Imbalance —
  e.g. 38 short passages and 2 mondai-6 — is a coverage gap.
- **Length match.** Mondai-4 passages 100-200 chars (kana-counted);
  mondai-5 passages 250-300 chars; mondai-6 passages structured as
  schedule/poster/notice with the question testing locate-the-fact
  rather than meaning.
- **Scope discipline.** Every kanji in the passage either in the N5
  whitelist (106 kanji) or in `data/dokkai_kanji_exception.json`
  (which must list a *reason*, e.g. "place name 京都" or "common
  signage 駅"). JA-28 enforces.
- **Topic variety — mandatory matrix.** The N5 corpus must cover
  the standard learner-life topic set:
  - Self-introduction (自己紹介)
  - Family / household (家族)
  - Daily routine (日常)
  - School / studying (学校・勉強)
  - Food / restaurant (食べ物・レストラン)
  - Shopping (買い物)
  - Transport / travel (交通・旅行)
  - Weather (天気)
  - Hobbies / leisure (趣味)
  - Health / body (体・病気)
  - Time / calendar (時間・カレンダー)
  - Work / occupation (仕事 — light at N5)
  Coverage thinner than 30 of 40 across these is a finding.
- **Question types — within passage.** For each passage, questions
  must mix:
  - Direct fact retrieval (passage explicitly states it).
  - Inference (light — "what does the writer plan to do tomorrow?").
  - Vocabulary-in-context (a word's meaning given context).
- **Choices kanji subset of passage (JA-20).** Every kanji in the
  4 choice options must appear somewhere in the passage. Otherwise
  the distractor can be eliminated without comprehension.
- **Cultural neutrality / avoidance.** Passages should avoid
  Japan-only cultural assumptions for non-Japanese-domiciled
  learners: no assumed familiarity with school clubs, regional
  cuisines, holiday names beyond the most common (お正月, クリスマス).
- **Accessibility of audio.** Every passage has a TTS audio file
  for listening-while-reading practice (711 MP3s; verify coverage).

**Listening / Chokai pedagogical depth (N5-specific)**

The official JLPT N5 chokai paper is structured as:
- **Mondai 1 (課題理解)**: "What does the speaker do next?" — 7
  items, ~30-40 sec each, conversation followed by "What will the
  woman do?".
- **Mondai 2 (ポイント理解)**: "What is the speaker's point?" — 6
  items, longer setup, focus on the *reason* or the *one detail*.
- **Mondai 3 (発話表現)**: Picture-based; given a picture, pick
  what to *say* — 5 items.
- **Mondai 4 (即時応答)**: One-line stimulus; pick the right
  response — 6 items.

Total: 24 items, 30 minutes. The app currently ships **40 listening
items**. Audit:

- **Mondai-type tagging.** Each item carries `mondai` ∈ {1,2,3,4}
  and `format_type` from a closed enum. ISSUE-016 closed this in
  v1.12.30. Verify every item has both fields and the enum is
  closed (no `null`, no free-text).
- **Per-mondai count.** Mondai 1: ≥7 items. Mondai 2: ≥6.
  Mondai 3: ≥5. Mondai 4: ≥6. Items beyond the floor are bonus.
  Imbalance is a finding (round-3 audit found mondai-4 = 0).
- **Voice variety — gendered + age.** A 24-item exam exposes
  learners to ≥4 distinct voices (typically 2 female + 2 male,
  spanning at least 2 age groups). Single-voice gtts is a
  major gap. VOICEVOX free tier (multiple speakers, fully offline-
  generable at build time) is the upgrade path.
- **Pacing match.** Official JLPT N5 chokai is roughly 200-220
  morae/minute (slower than native conversation, faster than
  beginner classroom). Audio rendered at gtts default may be
  too fast or too even. Random sampling + native-listener review
  required.
- **Background ambience.** Real exam audio includes occasional
  ambient context (train station hum, classroom murmur). Pure
  TTS-only audio sounds artificial. Adding low-volume ambience
  to mondai-1 and mondai-2 audio is a polish lever.
- **Script naturalness.** Listening scripts often contain
  fillers (あの、えー、まあ), self-corrections, false starts —
  these mark a script as authentic. Pure-textbook scripts read
  too clean.
- **Pre-listening preview.** Each item should have a `prompt_ja`
  shown *before* the audio (matching the official exam, where
  the question is read aloud before the dialogue).
- **Choices kanji subset constraint.** Same as dokkai (JA-20):
  the four choices' kanji must be subset of the script's kanji.
- **Repetition policy.** Real exam plays each audio once. The
  app should default to "play once" but allow on-demand replay
  in self-study mode (with a per-item replay counter for the
  progress dashboard — reveals where the learner struggles).

**Mock test (full-paper) authenticity (N5-specific)**

Real JLPT N5 paper structure:

| Section | Time | Items | Sub-mondai |
|---|---|---|---|
| 言語知識（文字・語彙） | 25 min | 35 | M1 漢字読み (12) / M2 表記 (8) / M3 文脈規定 (10) / M4 言い換え類義 (5) |
| 言語知識（文法）・読解 | 50 min | 32 | M1 文の文法 1 (16) / M2 文の文法 2 (5) / M3 文章の文法 (5) / M4 短文 (3) / M5 中文 (2) / M6 情報検索 (1) |
| 聴解 | 30 min | 24 | M1 課題理解 (7) / M2 ポイント (6) / M3 発話表現 (5) / M4 即時応答 (6) |

Pass mark: 80/180 total, with per-section minimums (38 / 19 / 19).

The app currently ships **28 mock-test papers** (402 paper-bound
questions). Audit:

- **Per-paper structure match.** Does each paper replicate the
  three-section, sub-mondai-tagged structure above? Or are papers
  flat 15-question sets that don't mirror the real exam?
- **Timing.** When the paper is launched in test mode, is the
  timer set to the official 25 / 50 / 30-minute splits, or to a
  generic single-timer?
- **Per-section minimums.** Does the scoring engine surface the
  per-section minimum-pass requirement (38/19/19), or only the
  total?
- **Score breakdown.** End-of-paper results should show: per-
  mondai correct/total, weighted score per section, total /180,
  pass/fail per section (not just total).
- **Question-type weighting in paper-builder.** When
  `tools/build_papers.py` slices questions into papers, does it
  weight each mondai per the official distribution? Or are
  questions distributed evenly?
- **Distractor quality at exam standard.** Distractors that a
  real JLPT N5 paper would NEVER use (e.g. obviously-wrong choices,
  N1-level kanji as distractors, non-existent verb forms) downgrade
  the paper's perceived authenticity.
- **Audio handling in 聴解.** In test mode, listening items must
  play exactly once with no replay control (real-exam fidelity);
  practice mode allows replay. The engine must enforce this.

**N5 scope discipline (cross-skill enforcement)**

- **Kanji whitelist compliance.** Stems and keyed answers use only
  the 106-kanji whitelist. Distractor exception is documented (per
  JA-13). All exceptions must appear in the augmented N5 set
  (`PRAGMATIC_N5_AUGMENTATION` in
  `tools/check_content_integrity.py`).
- **Vocabulary list compliance.** Keyed answers use only the N5
  vocab list. Slot-fill stems may use slightly extended vocab if
  documented.
- **Grammar N5-only in stems.** Flag any of these as N4+ leakage
  in N5 stems (not just questions, but ALL learner-target text):
  - Potential form (〜られる) outside the N5-borderline
    `〜ことができる` paraphrase.
  - ので outside specific late-N5 register-contrast pages.
  - 〜たことがある with specific time markers ("yesterday I have
    eaten" — ungrammatical).
  - Te-form chains of more than 2 verbs.
  - Causative (〜させる), passive (〜られる), causative-passive
    (〜させられる).
  - Conditional 〜たら / 〜ば in places where 〜と would be natural
    at N5.
- **Listening scope.** Scripts use N5 vocab + grammar only. Pace
  matches official N5 (slower than native conversation).
- **Reading scope.** Per JA-28 + the dokkai kanji exception list.

**Corpus structural properties**

- **Answer-position distribution.** Per corpus, the correct-answer
  position should be balanced 25/25/25/25 ±2 across positions 1-4.
  Drift indicates either lazy authoring (always answer-3 syndrome)
  or builder bias (papers slicing in fixed positions).
- **Sub-section balance.** Within a corpus (e.g. moji), the
  per-mondai sub-counts should match the JLPT-official ratios.
- **Duplicate stems.** No stem (after stripping HTML / format /
  punctuation) appears in more than one item (JA-7 invariant).
- **Duplicate target words.** If 5 questions all test 食べる, the
  distinct-vocab coverage is poor.
- **Near-identical choice sets.** Different stems with the same
  4-option choice set is suspicious — usually a copy-paste bug.
- **Mirror-pair adjacency.** Vocab pages for transitive/intransitive
  pairs should sit adjacent. Same for honorific/plain pairs.
- **Coverage breadth.** Every N5 kanji (106) appears in at least
  one example sentence and at least one question. Same for
  every vocab cluster (1041 entries) and every grammar pattern
  (178). A "ghost" item — present in catalog but never tested —
  is a coverage gap.

**L1-interference and learner-language coverage (Hindi-focused)**

This dimension is the strongest single content lever for niche-N1
(native-Hindi-medium JLPT prep).

The single target L1 is **Hindi**. Audit:

- **Locale completeness — content surfaces.** Beyond UI chrome,
  what fraction of grammar `meaning_en`, vocab `gloss` →
  `gloss_translations.hi`, kanji `meanings` →
  `meanings_translations.hi`, common-mistakes `wrong/right/why`
  blurbs, reading `explanation_en`, listening `explanation_en`
  are translated to Devanagari Hindi? Today the post-transition
  baseline is approximately 100% en + 0-X% hi (X depends on the
  Phase 1 seed translation pass; verify by reading the actual
  hi.json + per-entry `*_translations.hi` populations). Moving
  hi coverage to ≥80% on every content surface is THE niche-N1
  outcome.
- **L1-interference content presence (Hindi-specific notes).**
  For each grammar pattern, is there a Hindi-L1 note covering
  the relevant subset of:
  - Postposition→particle mapping (से→から/で, को→を/に, में→に/で)
  - Verb-agreement transfer (Hindi gender/number → none in JP)
  - Tense over-marking (Hindi multi-tense → JP non-past/past)
  - Politeness mismatch (3-tier Hindi pronoun → JP morphological
    keigo)
  - Negative-formation placement (नहीं at start vs ない attached)
  - Question-particle position (क्या first vs か last)
  - Plural marking (Hindi marks vs JP usually doesn't)
  - Counter-system overlap (familiar concept, unfamiliar inventory)
  - SOV order shared advantage (note as confidence builder)
  Patterns where these contrasts apply MUST carry the Hindi note.
  Patterns where the contrast does not apply may omit. Generic
  "non-English speakers benefit from..." text is a finding —
  Hindi notes must be Hindi-specific.
- **Kanji-cognate awareness (cross-cutting English-language note).**
  Hindi has no shared script with kanji, so the Mandarin-style
  shared-glyph cognate trap does not apply to a Hindi-L1 learner.
  HOWEVER: a non-trivial fraction of the Indian learner audience
  has prior Mandarin exposure (business Chinese, anime/manga
  vocabulary picked up via Chinese fan-subs). For those learners,
  the same false-friend warnings (勉強 ≠ 勉强, 大丈夫, 手紙 ≠ 手纸,
  娘) apply — surface them as English-language pedagogical notes
  on the kanji's English meaning panel, NOT as
  `meanings_translations.hi` content.
- **Romanization conventions.** When a vocab entry is shown to a
  learner who hasn't yet learned the kana for it, the fallback
  romanization must be consistent across both en and hi locales:
  - **Hepburn romanization** is the project standard. This is
    the convention most Japanese-language textbooks worldwide
    use, including Indian university Japanese-language courses
    and Yoisho Academy.
  - Devanagari transliteration of Japanese is NOT used. The
    Hindi locale displays Hepburn romaji for the romanized
    fallback, and Devanagari Hindi only for the gloss/meaning.
    This keeps the audit invariant simple.
  Verify the convention is consistent across the app.
- **Native-speaker review of Hindi content.** Hindi translations
  and L1-interference notes should be reviewed by a native Hindi
  speaker (not just LLM-curated). Today's post-Phase-1 baseline
  of the locale transition is ~100% `review_status: llm_curated`
  on Hindi content. Flag every Hindi-locale entry as
  `review_status: llm_curated` until a native-review pass
  completes. The native-review pipeline cost / sourcing /
  schedule is an open question (see Section 6).

## PRIORITIZATION

Every entry must carry both a SEVERITY and a PRIORITY.

**Severity** — what kind of thing is this:
- BLOCKER: incorrect content shown to users, broken core feature,
  privacy-contract violation, or factually wrong information that
  damages learner trust.
- MAJOR: gap from best-in-class OR from a defensible niche position
  that meaningfully reduces the app's value or saleability.
- MINOR: rough edge that's noticeable but doesn't block use or
  meaningfully degrade outcomes.
- IMPROVEMENT: not broken; would raise the ceiling toward best-in-niche.

**Priority** — what should be done first, on a 1–5 scale:
- P1 (do first): high-impact AND ship-blocking AND inexpensive to
  fix. Fixes obvious correctness or trust issues. No reason to defer.
- P2 (do soon): high-impact, may be larger lift but clearly worth
  doing in the current cycle. Closes major gaps from best-in-niche or
  unlocks a niche claim.
- P3 (do this cycle): meaningful improvement, lower urgency, may
  depend on P1/P2 work landing first.
- P4 (next cycle): nice-to-have; defer unless cheap to land alongside
  related work.
- P5 (someday/maybe): aspirational or low-impact; document but don't
  schedule.

Priority ≠ severity. A MINOR ID can be P1 if it's a one-line fix that
removes user-visible drift (e.g., the localStorage namespace doc
mismatch). A MAJOR can be P3 if it requires foundational work first.
Do not collapse them into one number.

For every entry also assign IMPACT (HIGH/MEDIUM/LOW) and EFFORT
(HIGH/MEDIUM/LOW), and a NICHE-FIT tag (N1/N2/N3/N4/none) marking
which niche claim the item supports (or "none" if it's pure
correctness work). Priority should be defensible from
severity + impact + effort + niche-fit + dependencies.

## REGISTRATION (the one allowed write)

This audit is otherwise read-only. There is exactly one write
permitted: registering the findings into the canonical issue
registry at:

  N5/feedback/n5-audit-2026-05-04.xlsx

This file is the single source of truth for open issues across
all N5 audit rounds (currently 138 rows, IDs ISSUE-001..055 +
IMP-001..079, with Decision values Done / Defer / Avoid). Findings
that live only in the audit transcript are lost across sessions
and have to be re-discovered next round — that is the failure mode
this section exists to prevent.

After producing Sections 0–7, you MUST register every Section-2
issue and every Section-3 improvement into the xlsx Items sheet,
following the existing 16-column schema (header at row 4):

  Column 1  ID                 (ISSUE-NNN or IMP-NNN)
  Column 2  Type               (Issue / Improvement)
  Column 3  Severity           (BLOCKER / MAJOR / MINOR / IMPROVEMENT)
  Column 4  Priority           (P1..P5)
  Column 5  Impact             (HIGH / MEDIUM / LOW)
  Column 6  Effort             (HIGH / MEDIUM / LOW)
  Column 7  Category           (from the audit dimensions)
  Column 8  Location           (file/route/item ID)
  Column 9  Title              (short, one line)
  Column 10 Current state      (concrete description)
  Column 11 Why-it-matters     (best-in-class comparison)
  Column 12 Suggested direction (one sentence)
  Column 13 Dependencies       (other IDs)
  Column 14 Decision           (Fix / Avoid / Defer)
  Column 15 Description        (long form / context)
  Column 16 Permission         (auto-action OK, or needs explicit user permission)

ID assignment rules:
- ISSUE IDs continue from the highest existing `ISSUE-NNN` in the
  sheet. Inspect the sheet first; do NOT reuse IDs.
- IMP IDs continue from the highest existing `IMP-NNN`.
- The audit must perform a one-shot read of the xlsx to discover
  the next free numbers. The pattern from
  `tools/register_dev_issue_list_deferrals_2026_05_05.py` is the
  reference implementation:
    `wb = openpyxl.load_workbook(XLSX, read_only=True)`
    iterate Items rows ≥ 5, collect existing IDs, take max(NNN)+1.

Decision-field defaults for new findings:
- **Fix** — action this cycle. Default for Section 2 P1/P2 issues
  and Section 3 P1/P2 improvements where the work is unblocked.
- **Defer** — has a precondition (N4 unblocked, native-review
  budget approved, design decision pending). The Description field
  must name the precondition.
- **Avoid** — explicitly NOT doing (Section 7 anti-items that the
  audit wants on the record so they don't resurface in future
  rounds). Width-additions in this depth-first cycle go here.

Permission-field defaults:
- "No permission required" — for content-depth fixes inside the
  data files the maintainer already controls.
- "Requires <X>" — name the gating decision concretely:
  e.g. "Requires native-reviewer engagement" / "Requires N4
  work-block lift" / "Requires monetization stance decision".

Registration mechanics — produce one Python script:

  N5/tools/register_audit_<YYYY_MM_DD>.py

modeled on `tools/register_dev_issue_list_deferrals_2026_05_05.py`.
The script must:

2. Build a `set` of existing IDs from rows ≥ 5 of the Items sheet

3. Iterate the in-script `NEW_ROWS` list of tuples.
4. Skip-and-warn on any ID already in the sheet (idempotent re-run).
5. Append the rest at `ws.max_row + 1`.
6. Save the workbook.
7. Print appended/skipped counts.

  commit script + xlsx together. No other writes anywhere.

  the feature.
