from collections import Counter
pat_miru  = re.compile(r"^[^\s]{1,10}を\s*見ました(?:よ|ね|よね)?。?$")
pat_doko  = re.compile(r"^あの\s+(.{1,10})は\s*どこですか。?$")

pat_quote = re.compile(r"^「(.{1,10})」と\s*あいさつしました。?$")
GREETINGS = {"おはよう","こんにちは","こんばんは","ありがとう","さようなら",
             "おやすみ","ただいま","おかえり","もしもし","いってきます",
             "いってらっしゃい","おかげさまで"}
n_miru = n_doko = n_kore = n_greet = n_dup = 0
ja_xrefs = Counter()
for v in V:
    section = v.get('section','')
    pos = v.get('pos','')
    form = v.get('form','')
    is_loc = '13-locations' in section or '26-house' in section
    is_dem = pos == 'demonstrative' or 'demonstrative' in section.lower()
    exs = v.get('examples', [])

        if ja and ja in seen: n_dup += 1
        elif ja: seen[ja] = i
        ja_xrefs[ja] += 1

import re
PATTERNS = [
    re.compile(r'[をｦ]\s*(?:が\s*)?(?:大?好き|だいすき|すき|じょうず|上手|へた|下手|嫌い|きらい|ほしい|欲しい|わか(?:り|る|れ)|分か(?:り|る|れ))'),

    re.compile(r'(?<![ぁ-ゟ一-鿿])くないだ(?![け])'),
    re.compile(r'ました(?:した|ました)'),
    re.compile(r'ですです'),
    re.compile(r'(?<!ひとり)でに\s'),  # ひとりでに is a fixed adverb (FP-11)
    re.compile(r'にで\s'),
    re.compile(r'きれいい|げんきい|しずかい'),

## SATURATION DEFINITION (binding — supersedes prior "0 actionable" wording)

The audit is **saturated against its current pattern set** — and the
calendar block in line 1 fires — when ALL of these hold simultaneously:

1. **Phase-0 100%-rows count is unchanged from prior run.** Every
   row that was at 100% last cycle is still 100%; no row regressed.
2. **No row that was below 100% last cycle dropped further.** A
   row going from 88% → 80% is regression, not saturation.
3. **The qualitative "Section 5 + 6 actionable items" count is 0
   on two consecutive runs.** One zero-signal run is NOT saturation
   (could be sampling luck). Two consecutive zero-signal runs is.
4. **The CI invariants count in the CURRENT STATE table matches the
   live tools/check_content_integrity.py invariant count.** If they
   differ, the prompt has drifted and is not a reliable saturation
   detector.

If 1+2+3+4 all hold: the audit is closed *against the patterns this
prompt currently checks*. If ANY fails: the prompt is not yet
saturated and may be re-run when a trigger condition fires (top of
file).

**Important framing for the audit report:** "saturated" here means
*the patterns this prompt is defined to detect* return 0 findings on
the live corpus. It does NOT mean the corpus contains no defects of
any kind. New finding-classes may exist outside the patterns this
prompt currently names; they would surface only when a new pattern is
added to this prompt, when the prompt is re-run against an extended
corpus, or when native-human review surfaces an item the checker
doesn't catch. Always write "saturated against this prompt's pattern
set" in the report — never just "saturated" — to avoid implying
universal corpus correctness. (Cross-ref: WRITING DISCIPLINE FOR
AUDIT DOCS in `prompts/Japanese language Accuracy check.txt`.)

## LESSONS FROM PRIOR CYCLES (don't relitigate these)

These lessons came from 13 audit cycles producing ~185 content fixes
across 2026-04 through 2026-05-13. The audit MUST respect them; doing
so prevents loops where each new cycle re-discovers the same gap.

### Anti-pattern A: ad-hoc sampling at saturation produces false positives
Run-3 of 2026-05-13 (under the prior version of THIS prompt) "found"
several issues that did not exist on inspection — they came from
inspecting 6 random items and pattern-matching against memory of run-1
findings. The fix: Phase-0 mechanical checklist (Step 1.6 above)
produces the actual numbers; qualitative findings are layered on top
of mechanical numbers, never instead of them.

### Anti-pattern B: human-day effort estimates are invalid
The audit MUST NOT estimate effort in human-person-days. The audit
runs in Claude, the fixes are applied by Claude, the reviewer is one
human. Use Claude-turns: "1 turn", "2-3 turns", "5+ turns". A turn
is a single Claude response cycle. This was added to CLAUDE.md as
anti-pattern #22.

### Anti-pattern C: calendar-cadence audit reruns produce zero signal at saturation
The DO-NOT-RUN-ON-CALENDAR block at line 1 of this file exists
because three runs in one working session produced 29 / 5 / 0
actionable findings. The third run was confirmatory only and added
zero value. Calendar reruns (every Friday, every release, every PR)
spend reviewer attention on confirmatory output. Run ONLY on trigger
conditions (see line 1).

### Anti-pattern D: scorecard cells calcify against the live corpus
The 2026-05-13 cycle found 9 wrong field-name lookups producing
false-0% readings. The cells in this prompt represent what the
authors EXPECTED the field-names to be; the live data is the source
of truth. Step 1.5 (Field-Name Verification) fires automatically
before any "below bar" claim.

### Anti-pattern E: width-additions are out of scope for richness audits
Every cycle, the temptation is "let's add 20 more vocab to fill
gaps." The RICHNESS-FIRST mandate (line ~200) freezes widths.
Depth on EXISTING entries is the work. If the audit produces "add
N more vocab" as a P1 finding, the prompt has lost the richness-
first discipline and should NOT be acted on without explicit
maintainer override.

### Anti-pattern F: provenance tier is not a user-facing trust signal
The `native_reviewed` provenance tag is a TRACKING LABEL ONLY —
nothing in the corpus has been reviewed by a Japanese native speaker;
the maintainer hand-curated and LLM-promoted entries carry this tag.
Exposing the tag in UI as a trust signal would be a false claim.
IMP-184 was set to Avoid 2026-05-13 for this reason. Do not surface
provenance to learners; the content's correctness is the trust signal.

### Anti-pattern G: absolutist phrasing in audit reports (added 2026-05-15)
Audit report prose must NOT use terminal/conclusive words that imply
universal coverage. Specifically banned in the report's section
headings, summary tables, and conclusions:

  - "every X" / "all X" without scope qualifier
  - "0 findings" without naming the patterns scanned
  - "RESOLVED" / "complete" / "final" without scope qualifier
  - "closed enum" without "against currently-observed values"
  - "JA-NN locks the gain" — instead "JA-NN prevents re-introduction
    of these specific named patterns"
  - "saturated" / "converged" without "against this prompt's pattern
    set"
  - "Final CI count" — instead "CI invariants at this checkpoint"
  - Statistical-extrapolation claims from sub-2% samples
    ("≥95% likely natural from a 80-sample n=6,300 corpus") — the
    sample shows 0 findings, but extrapolation is not justified

Why: future readers of audit reports (native reviewers, institutional
adopters, learners) treat absolutist language as "the team verified
universal correctness." When a JLPT exam or native review surfaces a
defect, they conclude the report misrepresented coverage. The fix is
qualified writing — see WRITING DISCIPLINE FOR AUDIT DOCS in
`prompts/Japanese language Accuracy check.txt` for the rewrite table.

When THIS prompt's audit report is generated, the report MUST use the
qualified phrasing throughout. Run a final pre-commit grep on the
report against the banned-word list; rewrite any hit.

You will NOT fix anything. You will NOT implement anything. You

will produce one ranked list and stop.

## STRATEGIC FRAMING (read this first)

The JLPT prep market has well-funded incumbents. Each owns one
core dimension:

  - **Bunpro** — native-reviewed grammar with 5-7 example
    sentences per pattern, textbook-aligned grammar paths
    (Genki I L1 → L2 → ...), self-grading reviews with JP-keyboard
    input + 50% partial credit, ghost reviews (failed cards
    reappear sooner).
  - **WaniKani** — kanji with 3-mnemonic structure (radical-story
    + visual + reading mnemonic), SRS gating (can't see vocab
    until kanji is learned), lifetime users, paced 25-30 min/day
    commitment.
  - **Renshuu** — mobile-first multi-skill drills, custom lists,
    community decks.
  - **Tofugu** — pedagogical-depth essays per pattern (longer than
    any textbook), free.
  - **Migaku** — sentence-mining from authentic media, Chrome
    extension, Anki deck export.
  - **JapanesePod101** — story-based audio dialogues with native
    voice talent, cultural notes per lesson.
  - **Anki + shared decks** — free SRS.
  - **JLPT Sensei** — free web reference; thin per-pattern
    explanations + one example each.

Trying to beat these on their CORE game is a losing audit
direction. THE AUDIT MUST OPTIMIZE FOR CONTENT-DEPTH DIMENSIONS
WHERE THIS APP CAN CREDIBLY OUT-COMPETE THEM, plus dimensions
where NONE of them deliver well.

JLPTSuccess's existing defensible niches:

  N1. **Privacy / no-account / offline / ad-free.** Bunpro requires
      account + subscription; WaniKani requires account; Renshuu
      has ads in the free tier; Duolingo gamifies + ads.
      JLPTSuccess runs fully on-device, no login, no telemetry.
  N2. **Open content for institutional / self-hosted use.**
      Vocational schools, language schools, NGOs could fork and
      self-host. Is the project structured to make this viable?
  N3. **All-in-one.** Single app for grammar + vocab + kanji +
      reading + listening + test mode. Is every surface deep
      enough that a learner doesn't have to leave for that skill?

The audit's saleability target is: **"the only privacy-first,
no-account, offline JLPT N5 app where every content surface is
demonstrably richer than the dedicated incumbent on that
surface."** That's a defensible niche. "World-class generic JLPT
app" is not.

## INTERCONNECTION-DENSITY CHECK (a first-class richness dimension)

Content richness = per-entry depth × cross-reference density. The
audit must produce these numbers:

```
Density-1: every grammar pattern → how many vocab entries link IN via vocab_ids? (target: ≥10 average; ≥3 floor per pattern)
Density-2: every vocab entry → how many grammar patterns reference it? (target: ≥2 average; ≥1 floor per word)
Density-2b (REVERSE): every vocab entry → which patterns it FREQUENTLY appears with (e.g., 食べる → ~たい, ~たことがある, ~ながら). This is a different field from "patterns that reference this word" — Density-2 is grammar→vocab; 2b is vocab→pattern co-occurrence. (target: ≥3 patterns per high-frequency content word)
Density-3: every kanji → how many vocab entries use this kanji? (target: ≥5 average; ≥2 floor per kanji)
Density-4: every kanji → how many reading passages contain this kanji? (target: ≥3 average; floor ≥1 per kanji)
Density-5: every reading passage → how many sentences have inline grammar-footnote links? (target: 100% of sentences)
Density-6: every listening item → vocab in transcript clickable to vocab page? (target: 100%)
Density-7: every paper question → does it cross-link to its grammar/vocab/kanji entry on review? (target: 100%)
Density-8: every confusable kanji cluster → all members linked to all others? (target: 100%)
Density-9 (NEW): every grammar pattern with register variants → does it cross-link to the politeness ladder (casual/polite/humble/respectful chain)? (target: 100%)
Density-10 (NEW): every vocab entry with pragmatic-multi-function → does it enumerate all functions inline? (target: 100% of multi-function entries)

Aggregate metric: total cross-references / sqrt(corpus size)
This number is the single most underrated quality metric.
A learner-facing app where every entry is a dead-end is "shallow"
even if each entry is itself well-authored.
```

The audit must report each density number plus the aggregate. The
priority of fixes follows from which density has the worst gap.

## PEDAGOGICAL CORRECTNESS (native-teacher dimensions)

These dimensions are what an experienced N5 classroom teacher
would catch but a UI-focused audit wouldn't. Each is a distinct
audit area; each item below maps to its own scorecard row above
or to a Section-5 / Section-6 finding.

### Particle-precision audit (は vs が is the #1 error source)

For every grammar example and question stem using は or が,
audit must verify:
- Information-structure annotation: is the topic GIVEN
  (は-marker) or NEW (が-marker)?
- Predicate type compatibility (state predicates → が-bias for
  experiencer; action predicates → は-bias for topic)
- "Exhaustive listing" reading of が is flagged where applicable
- 〜は〜が double-particle constructions explicitly explained

### Verb-class & inflection trap audit

- Group-1 exception verbs (X-6.6): 入る/帰る/走る/知る/切る/要る +
  Bunpro lists 喋る but it's late-N5+. Each must carry the
  `group1_exception: true` flag AND have its negative + te + ta
  forms tested in at least one item to verify the exception is
  honored, not just labelled.
- Irregular te-form: 行く → 行って (NOT 行いて). Audit must verify
  every 行く-related example uses the correct te-form.
- する-compound verbs vs lexical する: 勉強する is composed
  (べんきょう + する); 関する is lexical. Different conjugation
  patterns; audit per-entry.
- 来る reading shifts: 来る (くる), 来ます (きます), 来た (きた),
  来て (きて), 来ない (こない). The kun reading shifts by form;
  audit confirms each form's reading is correctly transcribed.

### Onomatopoeia / mimetic words (擬音語/擬態語)

Even at N5, real Japanese is mimetic-heavy. Audit must verify
the corpus includes at least these N5-level mimetics with full
schema (form, gloss, gloss_hi, examples, audio):
- ぺこぺこ (very hungry), にこにこ (smiling),
- どきどき (heart pounding), わくわく (excited),
- ぴかぴか (sparkling/clean), ゆっくり (slowly),
- ちょっと (a little — pragmatic), だんだん (gradually),
- もっと (more), まあまあ (so-so).

If the corpus has 0/10 of these, that's the largest leverage
gap on the vocab surface.

### Politeness ladder explicit teaching

For each grammar pattern with register variants, audit verifies
the ladder is shown EXPLICITLY (not just "casual / polite" tag):
- Causation ladder: から → ので → なので → ですので → でございますので
- Existence ladder: いる → いらっしゃる → おる
- Eating ladder: 食べる → 召し上がる → いただく
- Saying ladder: 言う → おっしゃる → 申す
- Going/coming ladder: 行く・来る → いらっしゃる → 参る

Native teachers explicitly show the ladder; textbooks underplay
it. Audit flags any pattern with register variants but no
ladder.

### Pragma-multi-function annotation (single-gloss is wrong)

Audit verifies every entry below has its multiple speech-act
functions glossed inline:
- すみません = {apology, attention-getter, gratitude-when-receiving}
- 大丈夫 = {confirmation, polite-refusal — "I'm OK, no thanks"}
- どうぞ = {offering, please-do, by-all-means}
- どうも = {thanks-shortened, sorry-shortened, hello-shortened, very}
- ちょっと = {a-little, pragmatic-softener "ちょっと…"-decline}
- 結構 = {fine, no-thanks-polite-refusal}

Single-translation glosses miss the function. Audit flags any
of these entries that doesn't enumerate functions.

### Phonological discrimination (listening / pitch / vowels)

## TEST-TAKING STRATEGY (受験対策) — JLPT-specific layer

Real JLPT prep books include a strategy layer that no incumbent

### Per-mondai timing strategy

Test-mode timer shows an internal pacing recommendation per

- 言語知識（文字・語彙）25min: M1 漢字読み (5min) / M2 表記 (5min) /
  M3 文脈規定 (10min) / M4 言い換え類義 (5min)
- 言語知識（文法）・読解 50min: M1 文の文法1 (15min) / M2
  文の文法2 (5min) / M3 文章の文法 (5min) / M4-6 読解 (25min)
- 聴解 30min: fixed pace by audio (no per-mondai discretion)

### Trap-pattern catalog (引っかけパターン)

Audit verifies the corpus / explanation surfaces document the
common JLPT N5 traps:
- 似た意味 (similar meaning) distractors in 言い換え類義 — MUST
  ship at least 5 worked examples with explicit "this looks
  right because... but is wrong because..." analysis
- 紛らわしい数字 (confusing numbers) in 聴解 — 七時 vs 一時 in
  fast speech, 4日 (yokka) vs 8日 (youka) discrimination
- 否定+否定=肯定 traps — N5-level double-negatives like
  「行かなくちゃいけない」 (must go) misread as "don't have to go"
- 順番引っかけ (sequence trick) in 課題理解 — speaker mentions
  multiple actions, only the last is the answer
- 漢字読み trap — same kanji compound has multiple readings;
  context disambiguates (一日 = いちにち / ついたち)

### Test-taking technique surfaces

- 消去法 (elimination): explicit "rule out two clearly wrong,
  pick between the remaining two" walk-through on a sample
  question per mondai type.
- 自信を持って先に進む (move on confidently): per-question
  budget reminder; flag-and-return-later UX.
- マークシート練習 (answer-sheet drill): mock-test mode shows
  the question number explicitly to mimic the bubble-sheet
  rhythm.
- 試験前夜 / 当日 routine note: pre-exam day checklist (sleep
  schedule, breakfast, what to bring) — short cultural artifact
  matching what Japanese cram schools provide.

### Score-breakdown diagnostic

End-of-paper results MUST surface:
- Per-mondai correct/total (NOT just per-section)
- Weighted score per section (lang+reading combined / listening)
- Total / 180 and pass/fail per the TWO post-2010 minimums
- Weakest mondai-type with one-click drill into that mondai
- Pace analysis: time spent per mondai vs recommended

If any of these are missing, file as Section-5 issue with
N3 niche-fit (test-mode realism is a defensible niche).

### Diagnostic + weak-area drilling

Native teachers track: "you missed 3 questions on は/が, here
are 10 more." Audit verifies:
- /diagnostic route ACTUALLY diagnoses (places learner at a
  level within N5: early / mid / late) rather than just being a

- /missed route auto-drills the weak area (10 follow-up

## PRIORITIZATION

Every entry must carry SEVERITY and PRIORITY.

**Severity**:
- BLOCKER: incorrect content shown to users; broken core feature;

  damages learner trust.
- MAJOR: gap from best-in-class OR from a defensible niche

- MINOR: rough edge that's noticeable but doesn't block use.
- IMPROVEMENT: not broken; would raise the ceiling.

**Priority** (1-5):
- P1: high-impact, ship-blocking, inexpensive. No reason to
  defer.
- P2: high-impact, may be larger lift, clearly worth this cycle.
  Closes major incumbent gaps or unlocks a niche claim.
- P3: meaningful improvement, lower urgency, may depend on
  P1/P2.
- P4: nice-to-have; defer unless cheap alongside related work.
- P5: aspirational or low-impact; document but don't schedule.

Priority ≠ severity. A MINOR can be P1 if it's a one-line fix
that removes user-visible drift. A MAJOR can be P3 if it
requires foundational work first.
