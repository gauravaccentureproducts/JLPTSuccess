## STRATEGIC FRAMING (read this first)

The JLPT prep market is saturated. Bunpro owns grammar with native-
reviewed content + community + 5,000+ example sentences. WaniKani owns
kanji with mnemonics + community + lifetime users. Anki + shared decks
own free SRS. Renshuu owns mobile-first multi-skill. JLPT Sensei owns
free web reference. Marugoto owns Japan-Foundation-official.

Trying to beat any of those at their core game is a losing audit
direction. THE AUDIT MUST OPTIMIZE FOR DEFENSIBLE NICHES, NOT FOR
GENERIC PARITY.

JLPTSuccess's plausible niches (validate or invalidate during the
audit; do not assume any of these is already realized):

  N1. **Multilingual non-English-native learners.** Vietnamese,
      Indonesian, Filipino, Nepali, Hindi, Chinese-speaking JLPT
      candidates are millions of learners under-served by English-
      first apps. JLPTSuccess already ships en/vi/id/ne/zh locales
      — but are they native-quality? Are explanations / rationales
      / common-mistakes blurbs translated, not just UI chrome?
  N2. **No-account / offline / private / ad-free.** Bunpro requires
      account + subscription; WaniKani requires account; Renshuu has
      ads in the free tier; Duolingo gamifies + ads. JLPTSuccess
      runs fully on-device, no login, no telemetry. Is this
      MESSAGED visibly to a first-time visitor, or is it invisible?
  N3. **Open content for institutional / self-hosted use.** Vocational
      schools, language schools, NGOs in SE Asia could fork and self-
      host. Is the project structured to make this viable (license,
      docs, bundle size, no domain-locked assets)?
  N4. **All-in-one (vs juggling Bunpro+WaniKani+Anki).** Single app
      covering grammar+vocab+kanji+reading+listening+test mode. Is
      every surface actually production-quality, or are some shallow
      enough that a learner would still need a separate app for that
      skill?

The audit must answer, for each niche: does the app credibly own this
niche today? If not, what's the smallest set of changes that would
let it credibly claim this niche?

A "world-class generic JLPT app" is not a viable target for a solo
+ AI build. "Best free multilingual privacy-respecting JLPT N5 app"
IS a viable target. Audit accordingly.

## PRIORITIZATION

Every entry must carry both a SEVERITY and a PRIORITY.

**Severity** — what kind of thing is this:
- BLOCKER: incorrect content shown to users, broken core feature,
  privacy-contract violation, or factually wrong information that
  damages learner trust.
- MAJOR: gap from best-in-class OR from a defensible niche position
  that meaningfully reduces the app's value or saleability.
- MINOR: rough edge that's noticeable but doesn't block use or
  meaningfully degrade outcomes.
- IMPROVEMENT: not broken; would raise the ceiling toward best-in-niche.

**Priority** — what should be done first, on a 1–5 scale:
- P1 (do first): high-impact AND ship-blocking AND inexpensive to
  fix. Fixes obvious correctness or trust issues. No reason to defer.
- P2 (do soon): high-impact, may be larger lift but clearly worth
  doing in the current cycle. Closes major gaps from best-in-niche or
  unlocks a niche claim.
- P3 (do this cycle): meaningful improvement, lower urgency, may
  depend on P1/P2 work landing first.
- P4 (next cycle): nice-to-have; defer unless cheap to land alongside
  related work.
- P5 (someday/maybe): aspirational or low-impact; document but don't
  schedule.

Priority ≠ severity. A MINOR ID can be P1 if it's a one-line fix that
removes user-visible drift (e.g., the localStorage namespace doc
mismatch). A MAJOR can be P3 if it requires foundational work first.
Do not collapse them into one number.

For every entry also assign IMPACT (HIGH/MEDIUM/LOW) and EFFORT
(HIGH/MEDIUM/LOW), and a NICHE-FIT tag (N1/N2/N3/N4/none) marking
which niche claim the item supports (or "none" if it's pure
correctness work). Priority should be defensible from
severity + impact + effort + niche-fit + dependencies.
